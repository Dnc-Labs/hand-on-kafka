# Bài 3.1 — Avro & Schema Registry

> Ghi lại **nguyên văn** nội dung buổi học: lý thuyết + câu hỏi của học viên + câu trả lời. Không chỉnh sửa nội dung.

---

## 1. Đặt vấn đề: vì sao không dùng JSON cho event?

Ở Module 0–2 ta hay nghĩ "gửi message thì cứ JSON cho nhanh". Nhưng trong hệ event-driven **production**, JSON thuần có 3 vấn đề chí mạng:

**a) Không có schema bắt buộc (schema-less).**
Producer đổi field `amount` (number) → `amount` (string), hoặc xóa field `currency`. Consumer **không hề hay biết** cho tới khi… nổ ở runtime, trên prod, lúc 2h sáng. JSON không ai "gác cổng" việc đổi cấu trúc.

**b) Tốn băng thông & lưu trữ.**
JSON gửi kèm **tên field trong từng message**: `{"accountId":"...","amount":1000,"currency":"VND"}`. Một topic 100 triệu message/ngày thì lặp lại chữ `"accountId"` 100 triệu lần. Kafka lưu cả tháng → lãng phí khổng lồ.

**c) Không có kiểu dữ liệu chặt.**
`"amount": 1000` là int hay long? `"timestamp"` là epoch millis hay ISO string? JSON nhập nhằng, mỗi consumer tự đoán → bug.

---

## 2. Avro giải quyết thế nào?

**Avro** là một **binary serialization format** (Apache, sinh ra cho Hadoop, giờ là chuẩn de-facto trong hệ Kafka). Ý tưởng cốt lõi:

> **Tách schema ra khỏi data.** Data đi trên wire chỉ còn **giá trị nhị phân thuần** (không kèm tên field). Schema được lưu **một nơi riêng** và tham chiếu tới khi cần.

Ví dụ message `{accountId, amount, currency}`:
- **JSON**: `~60 bytes` (kèm tên field, dấu ngoặc, dấu nháy).
- **Avro**: `~15 bytes` (chỉ giá trị, theo đúng thứ tự schema định nghĩa).

Schema Avro được viết bằng **JSON** (file `.avsc`), ví dụ:

```json
{
  "type": "record",
  "name": "MoneyDebited",
  "namespace": "com.dnclabs.banking.events",
  "fields": [
    { "name": "accountId", "type": "string" },
    { "name": "amount",    "type": "long" },
    { "name": "currency",  "type": "string" }
  ]
}
```

→ Đây chính là **"hợp đồng"** mà producer và consumer cùng tuân theo.

---

## 3. Vấn đề mới: làm sao consumer biết schema để giải mã?

Nếu data trên wire chỉ còn nhị phân thuần (không kèm tên field), thì consumer **phải có schema** mới đọc được. Câu hỏi: lấy schema ở đâu?

- ❌ **Cách ngây thơ**: nhét nguyên schema vào mỗi message → quay lại tốn chỗ y như JSON, vô nghĩa.
- ✅ **Cách đúng**: dùng **Schema Registry**.

---

## 4. Schema Registry — "kho hợp đồng" trung tâm

**Schema Registry** (Confluent) là một service HTTP riêng (ta đã dựng ở Module 2, cổng `8081`) làm nhiệm vụ:

> Lưu trữ tất cả schema, cấp cho mỗi schema một **ID số nguyên (schema id) duy nhất, toàn cục**.

Cơ chế hoạt động (cực quan trọng, senior phải nắm):

**Khi producer gửi message:**
1. Producer có schema (vd `MoneyDebited` v1).
2. Avro serializer hỏi Schema Registry: *"schema này đã đăng ký chưa?"*
   - Chưa → đăng ký, Registry trả về **schema id** (vd `42`).
   - Rồi → Registry trả luôn id `42` (có cache, không gọi mỗi lần).
3. Serializer đóng gói message theo **wire format**:

```
[ 1 byte magic = 0x00 ][ 4 byte schema id = 42 ][ N byte Avro binary payload ]
```

→ Message trên wire **chỉ kèm 5 byte header** (1 magic + 4 id), KHÔNG kèm cả schema.

**Khi consumer đọc message:**
1. Đọc 5 byte đầu → biết `schema id = 42`.
2. Hỏi Schema Registry: *"cho tao schema id 42"* (cache lại).
3. Dùng schema đó **deserialize** payload nhị phân → object.

---

## 5. Subject — khái niệm hay nhầm nhất

Đây là chỗ **90% người mới hiểu sai**, bạn nắm kỹ:

- **Schema**: bản thân định nghĩa cấu trúc (`MoneyDebited` với các field).
- **Subject**: **một "không gian phiên bản"** (namespace versioning) trong Registry. Mỗi subject chứa **một chuỗi version** của schema và áp **compatibility rule** (Bài 3.2).
- **Schema id**: id toàn cục của một bản schema cụ thể.

Mặc định (chiến lược `TopicNameStrategy`), subject được đặt theo topic:

```
Topic: banking.account.debited
 → Subject cho value: banking.account.debited-value
 → Subject cho key:   banking.account.debited-key   (nếu key cũng dùng Avro)
```

Một subject `banking.account.debited-value` có thể có nhiều version:
```
banking.account.debited-value
 ├─ version 1 → schema id 42  (record có 3 field)
 ├─ version 2 → schema id 57  (thêm field "description")
 └─ version 3 → schema id 63  (...)
```

→ **Subject = trục thời gian tiến hóa của schema cho một topic.** Bài 3.2 sẽ học các rule khống chế sự tiến hóa này (backward/forward compatibility).

---

## 6. Bức tranh tổng thể (ghi nhớ)

```
   PRODUCER                  SCHEMA REGISTRY (:8081)              CONSUMER
   --------                  ----------------------              --------
   object MoneyDebited
        │ serialize
        ├──"schema này id?"────────►  subject: ...-value
        │                             v1→id42, v2→id57 ...
        ◄──── id = 42 ────────────────┘
        │
   [0x00][42][binary] ──► Kafka topic ──► [0x00][42][binary]
                                                  │ đọc id=42
                                                  ├──"cho schema id 42"──►
                                                  ◄──── schema ──────────┘
                                                  │ deserialize
                                                  object MoneyDebited
```

---

📌 **Tóm tắt Bài 3.1 (phần lý thuyết):**
- **JSON event** có 3 điểm yếu prod: không ràng buộc schema, tốn chỗ, kiểu lỏng lẻo.
- **Avro** = binary format, **tách schema khỏi data**; data trên wire chỉ còn giá trị nhị phân thuần.
- **Schema Registry** = kho schema trung tâm, cấp **schema id**; wire format = `magic(1) + schema id(4) + payload`.
- **Subject** = trục version schema cho một topic (mặc định `<topic>-value`), nơi áp compatibility rule.

# Hỏi đáp 3.1 (nguyên văn)

## ❓ Câu 1: Schema Registry **chết** thì producer/consumer còn chạy được không?

Câu trả lời ngắn: **Có chạy được — NHƯNG có điều kiện, nhờ cache.** Đây là chỗ phân biệt người hiểu sâu với người học vẹt.

### Cơ chế cache cứu cả hệ

Cả Avro serializer (producer) lẫn deserializer (consumer) đều có một **cache nội bộ trong JVM**: ánh xạ `schema ⇄ schema id`. Một khi đã hỏi Registry một lần, nó **nhớ luôn**, lần sau không gọi HTTP nữa.

→ Vì vậy ta phải chia làm **2 trường hợp**:

### TH1 — Schema đã được dùng trước đó (cache nóng) ✅
- **Producer**: đã gửi `MoneyDebited v1` (id 42) rồi → id 42 nằm trong cache. Registry chết, producer **vẫn gửi bình thường**, vì nó chỉ cần dán id 42 vào header, không cần hỏi ai.
- **Consumer**: đã đọc message id 42 rồi → schema 42 trong cache. Registry chết, consumer **vẫn deserialize bình thường**.

→ **Một hệ đang chạy ổn định (steady state), Registry sập trong vài phút thường KHÔNG gây gián đoạn.** Đây là điểm trấn an quan trọng.

### TH2 — Gặp schema/ID **chưa từng thấy** (cache miss) ❌
Đây là lúc nổ:
- **Producer** muốn gửi một schema **mới** (vd vừa deploy version thêm field) → cache chưa có → phải hỏi Registry để lấy id → Registry chết → **`SerializationException`, message gửi thất bại.**
- **Consumer** nhận message mang **schema id lạ** (id 57 nó chưa cache) → phải hỏi Registry lấy schema 57 → Registry chết → **deserialize thất bại**, message kẹt (tùy cấu hình error handler — Module 5 ta học DLT).

### 🎯 Hệ quả thực chiến (senior phải biết)

| Tình huống | Registry chết có sao không? |
|---|---|
| Hệ chạy ổn định, không deploy schema mới | **An toàn** (cache cứu) |
| Đang **rolling deploy** version schema mới | **Nguy hiểm** — producer mới gửi schema chưa đăng ký được |
| Consumer **mới scale-up** (instance mới, cache rỗng) gặp message id chưa cache | **Nguy hiểm** |
| **Cold start** toàn hệ (vừa khởi động, cache rỗng hết) | **Rất nguy hiểm** |

### Cách prod xử lý (đây là phần "triển khai thực tế")
1. **Schema Registry phải chạy cluster (HA)** — tối thiểu 2–3 instance sau load balancer, **không để 1 node duy nhất**. (Lưu ý: ở Module 2 ta dựng **1 node Schema Registry** — đây là điểm **local/dev**, prod thật phải HA. Tôi flag ra để bạn ý thức.)
2. **Đăng ký schema lúc CI/CD, không phải lúc runtime.** Chiến lược prod: dùng `avro-maven-plugin` / `schema-registry-maven-plugin` để **register schema ở bước build/deploy**, không để producer tự-đăng-ký-lần-đầu lúc chạy thật. → Khi app khởi động, schema đã có sẵn trên Registry rồi. (Ta sẽ làm điều này ở Bài 3.3.)
3. **Tắt auto-register trên prod**: đặt `auto.register.schemas=false` cho producer prod → producer **không được phép** tự tạo schema mới, buộc mọi schema phải qua pipeline duyệt. Tránh việc một dev vô tình deploy code đẩy schema rác lên Registry.

> **Khẩu quyết:** *Registry không nằm trên đường đi (hot path) của từng message — nó chỉ bị chạm khi cache miss. Nhưng đừng ỷ lại cache: HA cho Registry + đăng ký schema từ CI/CD là bắt buộc ở prod.*

## ❓ Câu 2: Vì sao mặc định subject theo topic? Khi nào đổi strategy?

Schema Registry có **3 chiến lược đặt tên subject** (subject naming strategy). Hiểu được "tại sao" thì mới biết khi nào đổi.

### Strategy 1 — `TopicNameStrategy` (MẶC ĐỊNH)
```
subject = <topic>-value   (và <topic>-key nếu key dùng Avro)
```
**Triết lý:** *"Một topic = một loại event = một hợp đồng."*

- **Vì sao là mặc định:** đây là trường hợp phổ biến nhất và **an toàn nhất**. Compatibility check được áp **trên phạm vi topic** — Registry đảm bảo mọi schema từng xuất hiện trên topic `banking.account.debited` đều tương thích với nhau. Consumer của topic đó yên tâm: dù schema tiến hóa, nó vẫn đọc được (nhờ rule ở Bài 3.2).
- **Hạn chế:** ràng buộc **1 topic chỉ chở 1 loại record**. Muốn nhét nhiều loại event khác nhau vào chung 1 topic thì kẹt.

→ **Với hệ banking của ta, đây là lựa chọn đúng.** Mỗi topic (`account.debited`, `account.credited`, `fraud.checked`...) chở đúng một loại event. Ta sẽ dùng strategy này.

### Strategy 2 — `RecordNameStrategy`
```
subject = <namespace>.<record name>   (vd com.dnclabs.banking.events.MoneyDebited)
```
**Triết lý:** *"Subject theo loại record, KHÔNG quan tâm topic."*

- **Khi nào dùng:** khi bạn muốn **nhiều loại event đi chung một topic** (multi-type topic). Cùng một record type xuất hiện ở nhiều topic thì **dùng chung một hợp đồng**.
- Compatibility check áp **theo record type**, xuyên suốt mọi topic.

### Strategy 3 — `TopicRecordNameStrategy`
```
subject = <topic>-<namespace>.<record name>
```
**Triết lý:** lai 2 cái trên — *"nhiều loại event chung một topic, NHƯNG mỗi topic versioning độc lập."*

- **Khi nào dùng:** topic chở nhiều loại event, **và** bạn muốn cùng một record type được phép tiến hóa khác nhau ở từng topic.

### 🎯 Khi nào nên rời bỏ mặc định? — Tình huống kinh điển

> **"Tôi cần đảm bảo THỨ TỰ giữa nhiều loại event khác nhau của cùng một aggregate."**

Ví dụ rất sát domain của ta: một `Account` phát ra nhiều loại event — `AccountOpened`, `MoneyDebited`, `MoneyCredited`, `AccountFrozen`. Nếu mỗi loại đi một topic riêng, **không có gì đảm bảo consumer thấy chúng đúng thứ tự thời gian** (Bài 1.1: Kafka chỉ đảm bảo thứ tự **trong một partition**, không xuyên topic).

→ Giải pháp: cho **tất cả event của Account đi chung một topic** `banking.account.events`, **key = accountId** (cùng key → cùng partition → đúng thứ tự tuyệt đối cho một account). Nhưng giờ topic chở **nhiều loại record** → `TopicNameStrategy` mặc định **không dùng được** (nó cấm trộn record type) → buộc đổi sang `RecordNameStrategy` hoặc `TopicRecordNameStrategy`.

Đây chính là pattern **"event per aggregate"** (rất hay gặp trong **Event Sourcing**). Ta sẽ cân nhắc nó ở Module 6 khi làm Saga/CQRS.

### Bảng quyết định nhanh

| Nhu cầu | Strategy |
|---|---|
| 1 topic = 1 loại event (mặc định, an toàn) | `TopicNameStrategy` ✅ |
| Nhiều loại event chung 1 topic, chung hợp đồng toàn cục | `RecordNameStrategy` |
| Nhiều loại event chung 1 topic, version độc lập từng topic | `TopicRecordNameStrategy` |

> **Khẩu quyết:** *Mặc định `TopicNameStrategy` vì "1 topic 1 loại event" vừa phổ biến vừa cho compatibility check chặt nhất. Chỉ đổi khi buộc phải nhồi nhiều loại event vào chung một topic — điển hình là để giữ thứ tự xuyên-loại cho cùng một aggregate (event sourcing).*

## ❓ Câu 3: Subject là gì? Scope nó thuộc Kafka hay Avro?

Câu trả lời dứt khoát:

> **Subject KHÔNG thuộc Kafka, cũng KHÔNG thuộc Avro. Nó là khái niệm của riêng Schema Registry.**

Đây là chỗ làm bạn rối, nên ta tách bạch 3 tầng — chúng là **3 thế giới độc lập**:

```
┌─────────────────────────────────────────────────────────────┐
│  1. AVRO (thư viện serialization)                            │
│     - Chỉ biết: "schema" và "cách biến object ↔ binary".    │
│     - KHÔNG biết Kafka là gì. KHÔNG biết subject là gì.      │
│     - Bạn có thể dùng Avro để ghi file ra đĩa, không cần Kafka.│
├─────────────────────────────────────────────────────────────┤
│  2. KAFKA (broker — cái cluster 3 node ta dựng)             │
│     - Chỉ biết: topic, partition, offset, BYTES.            │
│     - Với Kafka, message chỉ là một mảng byte vô nghĩa.     │
│     - KHÔNG biết Avro, KHÔNG biết schema, KHÔNG biết subject.│
│     - Broker KHÔNG validate nội dung message bao giờ.        │
├─────────────────────────────────────────────────────────────┤
│  3. SCHEMA REGISTRY (server riêng, cổng 8081)              │
│     - Đây mới là nơi khái niệm "SUBJECT" tồn tại.          │
│     - Subject = một "ngăn tủ" chứa lịch sử version schema.  │
│     - Áp compatibility rule trên từng ngăn (subject) đó.    │
└─────────────────────────────────────────────────────────────┘
```

**Điểm chốt cần khắc cốt:** Kafka broker **hoàn toàn không biết** message của bạn là Avro hay JSON hay ảnh mèo. Với broker, mọi message chỉ là `byte[]`. Toàn bộ chuyện schema/subject/compatibility diễn ra **bên ngoài Kafka**, ở phía client (serializer) và ở Schema Registry.

**Vậy subject scope là gì?** Subject là một **cái tên (string)** dùng làm khóa tra cứu trong Schema Registry. Dưới mỗi subject là một **chuỗi version** schema + một **compatibility rule**. Hết. Nó không "nằm trong" topic, không "thuộc về" partition. Nó chỉ là một entry trong database của Schema Registry.

Sở dĩ ta hay nghe "subject = `<topic>-value`" là vì **chiến lược đặt tên mặc định** (`TopicNameStrategy`) chọn cách **đặt tên subject trùng tên topic** cho dễ quản lý. Nhưng đó chỉ là **quy ước đặt tên**, không phải Kafka ép buộc. Bạn hoàn toàn có thể đặt subject tên `con-meo-cua-toi` nếu muốn (đừng làm thế 😄). Đổi `RecordNameStrategy` thì subject lại đặt theo tên record, chẳng liên quan topic nữa.

> **Một câu để nhớ:** *Avro lo binary. Kafka lo bytes. Schema Registry lo subject + version + compatibility. Ba tầng tách rời — ghép lại nhờ 5 byte header trong wire format.*

## ❓ Câu 4: Avro buộc phải dựng thêm một server riêng — đúng không?

**Đúng. Đây là cái giá phải trả, và bạn nhận diện rất chuẩn.** Đánh đổi gồm:

1. **Thêm một thành phần hạ tầng** (Schema Registry) phải vận hành, monitor, **làm HA**, backup database schema, vá lỗi, nâng cấp.
2. **Thêm một điểm phụ thuộc** trong vòng đời message (dù chỉ chạm khi cache miss).
3. **Thêm độ phức tạp vận hành**: pipeline đăng ký schema, quản lý compatibility, governance.

Nói thẳng: với một **monolith** hay một dự án nhỏ 1–2 service, **chi phí này không đáng** — JSON thường là đủ. Avro + Schema Registry chỉ "đáng tiền" khi:
- Nhiều service (nhiều team) cùng đọc/ghi event.
- Event sống lâu (Kafka lưu cả tuần/tháng, hoặc compacted vĩnh viễn).
- Schema **chắc chắn sẽ tiến hóa** và bạn cần kiểm soát chặt để không phá consumer.

→ Hệ banking nhiều service, event là source-of-truth, đó là lý do ta chọn nó. Đây là **quyết định kiến trúc có chủ đích**, không phải "mặc định cho sang".

## ❓ Câu 5: Avro phải serialize/deserialize nên chậm hơn JSON, và "sao không dùng DTO chung cho producer/consumer tuân theo?"

Câu này có **2 hiểu lầm cần gỡ**, gỡ xong bạn sẽ thấy vì sao DTO chung **không giải quyết được vấn đề cốt lõi**.

### Gỡ hiểu lầm 1: "Avro serialize nên chậm hơn JSON"

> **Sai ở giả định gốc: JSON CŨNG phải serialize/deserialize.** Không có gì miễn phí cả.

Khi bạn gửi một object Java qua Kafka bằng JSON, Jackson phải:
- Producer: object Java → duyệt từng field → sinh chuỗi text `{"accountId":"...",...}` → encode UTF-8 → `byte[]`.
- Consumer: parse chuỗi text → đối chiếu tên field → reflection set vào object.

Đó **chính là serialize/deserialize**, và **JSON text parsing nói chung CHẬM hơn Avro binary**, vì:
- JSON phải parse text (tìm dấu `{`, `"`, `:`, escape ký tự...) — tốn CPU.
- JSON payload to hơn (~3–4 lần) → tốn băng thông, tốn I/O, tốn chỗ lưu trên broker.
- Avro đọc/ghi tuần tự theo schema cố định, không cần dò tên field → thường **nhanh hơn và gọn hơn**.

→ Thực tế Avro **thường nhanh hơn JSON**, không phải chậm hơn. Cái "chậm" duy nhất Avro thêm vào là **một lần hỏi Schema Registry lúc cache miss** — mà ta đã thấy là hiếm (chỉ lần đầu mỗi schema). Steady state thì Avro thắng cả tốc độ lẫn dung lượng.

### Gỡ hiểu lầm 2: "DTO chung sẽ thay được Schema Registry"

Đây là phần quan trọng nhất. Ý tưởng "DTO chung" thường là: **tạo một file JAR `common-dto` chứa class `MoneyDebited`, cả producer lẫn consumer cùng `import` nó.** Nghe rất hợp lý. Nhưng nó **sụp đổ trong hệ phân tán thật**, vì:

**a) DTO chung = coupling lúc COMPILE. Hệ phân tán cần tương thích lúc RUNTIME.**

Producer và consumer là **2 service deploy độc lập, vào 2 thời điểm khác nhau**. Kịch bản thực tế:
- Hôm nay: cả hai dùng `common-dto v1`.
- Tuần sau: producer cần thêm field `description`. Bạn sửa `common-dto → v2`, build lại producer, deploy producer.
- **Consumer vẫn đang chạy với `common-dto v1`** (chưa kịp deploy lại — và trong hệ thật, **không bao giờ deploy đồng thời mọi service được**).

Giờ producer (v2) gửi message có field mới. Consumer (v1) đọc bằng class cũ → **field mới biến mất hoặc nổ exception**. DTO chung **không cứu được**, vì nó chỉ đảm bảo "cùng class lúc biên dịch", **không đảm bảo gì lúc 2 phiên bản gặp nhau trên đường truyền**.

**b) Event sống lâu hơn code.**

Kafka lưu message cả tuần (hoặc compacted vĩnh viễn). Một message ghi bằng `v1` hôm nay, **6 tháng sau** một consumer mới (code `v5`) đọc lại để replay (Bài 1.4). Lúc đó class `v1` có khi đã bị xóa khỏi codebase. **DTO chung không có khái niệm "version cũ vẫn đọc được" — Avro + Schema Registry thì CÓ** (đó là Bài 3.2: schema id 42 vẫn nằm trong Registry, consumer mới vẫn lấy được schema cũ để giải mã).

**c) Không ai "gác cổng" thay đổi phá vỡ (breaking change).**

Với DTO chung, một dev đổi `amount: long → String`, build pass, deploy lên, **và phá toàn bộ consumer trên prod** — không ai cản. Với Schema Registry, khi producer cố đăng ký schema "đổi long thành string" mà subject đang để rule `BACKWARD`, **Registry TỪ CHỐI ngay lúc đăng ký** (`409 Conflict`), chặn đứng thảm họa **trước khi** nó lên prod. Đây là giá trị lớn nhất, và là cả Bài 3.2.

**d) DTO chung trói các service vào cùng ngôn ngữ + cùng nhịp release.**

JAR chung chỉ Java dùng được. Mai mốt có service Python/Go đọc cùng topic thì chịu. Avro là **chuẩn ngôn-ngữ-trung-lập** — Python/Go/Java đều sinh code từ cùng `.avsc`. Và mọi service không bị buộc phải nâng version JAR chung cùng lúc.

### Bảng đối chiếu

| Tiêu chí | DTO chung (JAR) | Avro + Schema Registry |
|---|---|---|
| Ràng buộc hợp đồng | Lúc **compile** | Lúc **runtime** (đúng nơi cần) |
| Producer/consumer lệch version | **Vỡ** | Vẫn tương thích (nếu theo rule) |
| Đọc lại event cũ sau nhiều tháng | Class cũ có thể đã mất | Schema cũ vẫn trong Registry |
| Chặn breaking change | Không ai chặn | Registry **từ chối lúc đăng ký** |
| Đa ngôn ngữ | Chỉ Java | Trung lập ngôn ngữ |
| Tốc độ/dung lượng wire | JSON to & chậm hơn | Binary gọn & thường nhanh hơn |

> **Khẩu quyết:** *DTO chung giải quyết "cùng cấu trúc lúc biên dịch". Vấn đề thật của hệ phân tán là "tương thích khi hai phiên bản gặp nhau lúc chạy, và khi đọc lại dữ liệu cũ". Chỉ schema có version + compatibility rule (Schema Registry) mới giải được — đó là toàn bộ lý do nó tồn tại.*

## ❓ Câu 6: Vậy tôi hiểu Avro làm serialize/deserialize (không để Java làm), và nó phải kết hợp với Schema Registry mới hoạt động được, không thì không hoạt động — đúng không?

Bạn tổng kết tốt, nhưng có **một ý đúng và một ý cần chỉnh lại cho chính xác**.

### ✅ Ý 1: "Avro làm serialize/deserialize thay cho Java" — ĐÚNG (cần nói chính xác hơn một chút)

Chính xác hơn: **Avro là một thư viện serialization thay thế cho cách serialize mặc định.** Khi gửi qua Kafka, bạn luôn cần biến `object → byte[]` (lúc gửi) và `byte[] → object` (lúc nhận). Câu hỏi chỉ là **dùng thư viện nào** để làm việc đó:

- Dùng **Jackson** → ra JSON text.
- Dùng **Java built-in Serializable** → ra binary kiểu Java (tệ, không nên dùng liên-service).
- Dùng **Avro** → ra binary theo schema.

→ Đúng, **Avro đảm nhận việc serialize/deserialize**. Nó không "ngăn Java làm", mà nó **là lựa chọn serializer bạn cắm vào** thay cho mặc định. Trong Spring Kafka, bạn sẽ cấu hình `value-serializer = KafkaAvroSerializer` (Bài 4.1) — chính là chỉ định "dùng Avro để serialize".

### ⚠️ Ý 2: "Avro phải có Schema Registry mới hoạt động, không thì không chạy" — **CHƯA ĐÚNG**

> **Avro hoàn toàn chạy được KHÔNG cần Schema Registry. Hai cái là 2 thứ TÁCH RỜI.**

Nhớ lại 3 tầng độc lập: **Avro** (thư viện serialize) và **Schema Registry** (server quản version) là **2 thế giới riêng**. Avro tự nó chỉ cần **một schema** là serialize/deserialize được. Bằng chứng:

- Avro được sinh ra cho **Hadoop**, để ghi file `.avro` ra đĩa — **chẳng có Kafka, chẳng có Schema Registry nào** ở đó cả.
- Trong file Avro thuần, **schema được nhúng thẳng vào đầu file**. Đọc file → có sẵn schema trong đó → giải mã được. Không cần hỏi server nào.

**Vậy Schema Registry sinh ra để giải bài toán gì?** → Bài toán **của riêng Kafka**:

Trong Kafka, mỗi message là độc lập và **rất nhiều** (hàng triệu/giây). Nếu nhúng nguyên schema vào **từng message** thì quay lại tốn chỗ y JSON (mục 3 — "cách ngây thơ"). Nên Confluent nghĩ ra: *thay vì nhúng cả schema, chỉ nhúng **schema id (4 byte)**, còn schema thật để ở một server tra cứu chung.* Server đó = **Schema Registry**.

→ Tức là: **Schema Registry là một TỐI ƯU cho việc dùng Avro TRONG Kafka, không phải điều kiện sống còn của Avro.**

### Phân biệt cho rõ — có 2 "kiểu Avro" khác nhau:

| | Avro thuần (không Registry) | Avro + Schema Registry (kiểu Kafka) |
|---|---|---|
| Schema lấy từ đâu? | Nhúng trong file / hai bên tự thỏa thuận sẵn | Tra trên server qua **schema id** |
| Wire format | Không có 5-byte header | Có `magic(1) + schema id(4) + payload` |
| Quản version, compatibility | **Không có** — tự lo | Registry lo (Bài 3.2) |
| Dùng khi | Ghi file, batch, RPC | **Streaming Kafka nhiều service** |

### Vậy trong DỰ ÁN của ta thì sao?

Vì ta dùng Avro **trong Kafka, nhiều service, schema sẽ tiến hóa** → ta chọn **kiểu thứ 2 (Avro + Schema Registry)**. Trong ngữ cảnh này thì **đúng là** producer/consumer của ta cấu hình `KafkaAvroSerializer` sẽ **cần** Schema Registry để chạy (vì nó dùng wire format có schema id, phải tra Registry — trừ lúc cache hit).

→ Nên câu của bạn **đúng trong phạm vi dự án này**, nhưng **sai nếu hiểu thành "Avro nói chung bắt buộc phải có Registry"**. Phân biệt được điều này mới là hiểu đúng bản chất.

> **Khẩu quyết:** *Avro tự thân chạy không cần Registry. Schema Registry là tối ưu để dùng Avro hiệu quả trong Kafka (gửi 4-byte id thay vì cả schema) + thêm quản trị version. Ta chọn cặp "Avro + Registry" vì ngữ cảnh đa-service, schema tiến hóa — nên trong dự án này hai cái đi cùng nhau.*

**Chỉnh đúng một câu:**

- ❌ "Avro phải có Schema Registry mới hoạt động."
- ✅ "Avro tự chạy được; nhưng **cách ta dùng Avro trong Kafka** (qua `KafkaAvroSerializer`) thì cần Schema Registry — vì ta chọn wire format dựa trên schema id."
