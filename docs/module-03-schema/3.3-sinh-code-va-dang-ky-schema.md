# Bài 3.3 — Sinh code từ Avro schema & đăng ký schema trong repo

> Ghi lại **nguyên văn** nội dung buổi học: lý thuyết + thực hành + kết quả thật. Đây là bài **code thật** đầu tiên của Module 3.

---

## 1. Lý thuyết: `avro-maven-plugin` hoạt động thế nào?

> **Bạn viết `.avsc` (schema). Maven plugin tự sinh ra class Java tương ứng lúc build. Bạn KHÔNG viết tay class event.**

Vì sao? Class Java và schema phải **luôn khớp tuyệt đối**. Nếu viết tay cả hai, lệch một field là vỡ. Cho máy sinh code từ schema → schema là **single source of truth**, class chỉ là sản phẩm phái sinh.

### Vòng đời build

```
  src/main/avro/MoneyDebited.avsc        ← BẠN viết (commit vào Git)
            │
            │  mvn generate-sources  (plugin chạy ở phase generate-sources)
            ▼
  target/generated-sources/avro/
     com/dnclabs/banking/events/MoneyDebited.java   ← MÁY sinh (KHÔNG commit)
            │
            │  mvn compile
            ▼
  target/classes/.../MoneyDebited.class
```

Hai quy tắc vàng:
1. **`.avsc` → commit vào Git** (contract, source of truth).
2. **Code Java sinh ra (`target/generated-sources`) → KHÔNG commit** (build artifact; `target/` đã trong `.gitignore`).

`generate-sources` là phase chạy **trước** `compile` trong vòng đời Maven (`validate → generate-sources → compile → test → package`), nên chỉ cần `mvn compile`/`package` là code tự sinh.

## 2. Cấu hình plugin trong `common-events/pom.xml`

```xml
<plugin>
  <groupId>org.apache.avro</groupId>
  <artifactId>avro-maven-plugin</artifactId>
  <version>${avro.version}</version>
  <executions>
    <execution>
      <phase>generate-sources</phase>
      <goals><goal>schema</goal></goals>
      <configuration>
        <sourceDirectory>${project.basedir}/src/main/avro</sourceDirectory>
        <outputDirectory>${project.build.directory}/generated-sources/avro</outputDirectory>
        <stringType>String</stringType>
        <fieldVisibility>PRIVATE</fieldVisibility>
        <enableDecimalLogicalType>true</enableDecimalLogicalType>
      </configuration>
    </execution>
  </executions>
</plugin>
```

3 dòng `configuration` đáng nhớ:
- **`stringType=String`**: sinh `java.lang.String` thay vì `CharSequence` (dễ dùng hơn nhiều).
- **`enableDecimalLogicalType=true`**: cho `decimal` → `java.math.BigDecimal`. **Cực quan trọng cho banking**.
- **`fieldVisibility=PRIVATE`**: field private, truy cập qua getter/builder.

## 3. ⚠️ Quyết định thiết kế: biểu diễn TIỀN TỆ

**Tuyệt đối KHÔNG dùng `float`/`double` cho tiền** (sai số dấu phẩy động). Hai cách đúng:

| | Cách A: `decimal` logical type → `BigDecimal` | Cách B: `long` lưu "đơn vị nhỏ nhất" |
|---|---|---|
| Khai báo | `{"type":"bytes","logicalType":"decimal","precision":18,"scale":2}` | `{"type":"long"}` |
| Map Java | `java.math.BigDecimal` | `long` |
| Ưu | Chính xác tuyệt đối, chuẩn đa-tệ | Đơn giản, nhanh, gọn |
| Hợp khi | Đa tiền tệ, độ chính xác kế toán | Chủ yếu VND, ưu tiên đơn giản/tốc độ |

→ **Quyết định đã chốt (qua AskUserQuestion): Cách A — `BigDecimal` qua `decimal(18,2)`** (chuẩn kế toán, an toàn nhất cho banking thật). Plugin đã bật `enableDecimalLogicalType=true` để khớp.

## 5. Schema đầu tiên: `MoneyDebited.avsc`

Event phát ra khi MỘT tài khoản bị **trừ tiền** (debit) trong luồng chuyển tiền. Producer: account-service. Topic: `banking.account.debited` (key = accountId).

Các field & quyết định kiểu:
- `eventId` (string) — UUID, cho idempotent consumer chống trùng (Module 5).
- `transferId` (string) — correlate các event cùng một Saga (Module 6).
- `accountId` (string) — tài khoản bị trừ; **cũng là message key** → cùng account vào cùng partition → thứ tự tuyệt đối (Bài 1.1).
- `amount` (decimal 18,2 → BigDecimal) — số tiền trừ.
- `currency` (string, **default "VND"**) — có default để dễ tiến hóa (Bài 3.2).
- `balanceAfter` (decimal 18,2) — số dư sau khi trừ; audit/dựng lại trạng thái.
- `occurredAt` (timestamp-millis → Instant) — thời điểm nghiệp vụ (epoch millis, UTC).

### Kết quả sinh code (đã verify)

`mvn -pl common-events generate-sources` → sinh `MoneyDebited.java`. Kiểm tra kiểu field sinh ra **đúng từng cái**:

| Field `.avsc` | Kiểu Java | Nhờ cấu hình |
|---|---|---|
| `eventId, transferId, accountId, currency` (string) | `java.lang.String` | `stringType=String` |
| `amount, balanceAfter` (decimal 18,2) | `java.math.BigDecimal` | `enableDecimalLogicalType=true` |
| `occurredAt` (timestamp-millis) | `java.time.Instant` | logical type Avro |

Class `extends SpecificRecordBase` + có **Builder** (`MoneyDebited.newBuilder()...build()`).

## 6. Bài tập học viên: `MoneyCredited.avsc` + review

Học viên tự viết event đối xứng (cộng tiền vào tài khoản nhận). **Review của mentor đã bắt các lỗi:**

1. 🔴 Typo tên topic trong `doc`: `banking.account.crebited` → sửa `credited`.
2. 🔴 `doc` field copy nhầm từ debit: *"Số dư SAU khi trừ"* → sửa *"SAU khi cộng/nhận tiền"*.
3. 🔴 Typo "Tài khoản nhận tiền **tiền**" (lặp chữ).
4. 🟡 **Bài học nhất quán contract:** học viên đặt tên field `accountBalance`, lệch với `balanceAfter` của `MoneyDebited` (cùng khái niệm). → Thống nhất `balanceAfter` cho cả hai. **Nguyên tắc: cùng khái niệm → cùng tên field xuyên suốt mọi schema.** (Và nhớ: tên field Avro **không nên đổi về sau** — đổi tên là breaking, Bài 3.2 → chốt tên chuẩn ngay từ đầu.)

Học viên sửa hết → generate lại sạch. Hai contract đầu tiên của hệ sẵn sàng.

## 7. Đăng ký schema lên Registry + set compatibility (governance Bài 3.2 → lệnh thật)

> ⚠️ **Điểm dev/local có chủ đích:** việc đăng ký dưới đây làm **thủ công bằng PowerShell một lần để học cơ chế**. Prod thật làm qua **CI/CD plugin** (mục 9). Đã flag cho học viên.

**Đổi cổng Schema Registry:** host `8081` → **`8061`** (máy học viên dùng 8081 nhiều). Chỉ đổi **cổng host** trong `docker-compose.yml` (`"8061:8081"`); các tham chiếu nội bộ Docker (`schema-registry:8081`, listener, healthcheck) **giữ nguyên** vì chạy trong mạng container. → Truy cập từ host qua `http://localhost:8061`.

**Bước 1 — set compatibility global = `BACKWARD_TRANSITIVE`** (đã chốt cho banking: đọc được toàn bộ lịch sử):
```
PUT http://localhost:8061/config   {"compatibility":"BACKWARD_TRANSITIVE"}
```

**Bước 2 — đăng ký 2 schema** → tạo subject:
| Subject | Schema id |
|---|---|
| `banking.account.debited-value` | (id cấp tự động) |
| `banking.account.credited-value` | (id cấp tự động) |

(Schema id chỉ cần **unique**, không cần liên tục — vài id bị "đốt" do lần ghi lỗi transient `50001 writing to Kafka store` lúc cluster vừa bật chưa ổn định ISR; retry là xong.)

### Bài học PowerShell (môi trường Windows)
Gửi schema qua REST trên PowerShell 5.1 vướng 2 quirk, đã xử lý:
1. `ConvertTo-Json` trên hashtable **bọc string thành `{"value":...}`** → dùng `[System.IO.File]::ReadAllText` (string thuần) + `ConvertTo-Json` trên **scalar** để escape, rồi ghép body thủ công.
2. **Encoding**: đọc/gửi phải UTF-8 (`ReadAllText` + `[Text.Encoding]::UTF8.GetBytes`) — nếu không, tiếng Việt thành mojibake (`phÃ¡t`).

## 8. 🛡️ Demo Registry chặn breaking change (phần đắt giá nhất)

Dùng endpoint **`POST /compatibility/subjects/<subject>/versions/latest?verbose=true`** để **TEST mà KHÔNG ghi** (đúng cách CI kiểm tra trước merge). Subject `banking.account.debited-value`, mode `BACKWARD_TRANSITIVE`:

| Ca thử | Kết quả | Giải thích |
|---|---|---|
| **A.** Thêm field `reason` **bắt buộc, không default** | 🔴 **CHẶN** | Registry trả lý do `READER_FIELD_MISSING_DEFAULT_VALUE`: *"field 'reason' không có default và thiếu trong schema cũ"* → reader mới không đọc nổi data cũ. **Đúng luật BACKWARD.** |
| **B.** Đổi `amount` `decimal` → `string` | 🟢 cho qua | **Bài học Avro tinh tế:** kiểu nền của `decimal` là `bytes`, mà Avro coi **`bytes ↔ string` promotable** → structural vẫn "tương thích". Registry kiểm theo **cấu trúc Avro**, KHÔNG theo ý nghĩa nghiệp vụ. |
| **C.** Thêm `description` **optional, default null** | 🟢 cho qua | Thêm field có default → BACKWARD an toàn. **Đúng luật.** |

**Hai kết luận then chốt:**
- Ca A = lá chắn governance thật: PR nào gây breaking, CI chạy `test-compatibility` sẽ **fail đỏ** trước khi lên prod.
- Ca B = **Registry không bắt được sai sót nghiệp vụ** (đổi `decimal→string` làm mất nghĩa tiền tệ nhưng vẫn lọt). → Vì vậy vẫn cần **code review + CI test**, Registry chỉ là một lớp.

> Lưu ý: phần `oldSchema` Registry trả về hiển thị mojibake tiếng Việt là do **terminal Windows**, không phải schema lưu sai. Schema trên Registry vẫn UTF-8 chuẩn.

(Trong lúc demo có lỡ ghi vài version rác vào subject → đã **hard-delete subject + đăng ký lại sạch** (chỉ còn version 1) trước khi chuyển sang test bằng endpoint không-ghi.)

## 9. 🏭 Tự động hóa — cách PROD làm (đã thêm vào repo)

Đăng ký/kiểm tra thủ công chỉ để học. Prod dùng **`kafka-schema-registry-maven-plugin`** chạy trong CI/CD. Đã thêm vào `common-events/pom.xml`:

```xml
<plugin>
  <groupId>io.confluent</groupId>
  <artifactId>kafka-schema-registry-maven-plugin</artifactId>
  <version>${confluent.version}</version>
  <configuration>
    <schemaRegistryUrls><param>${schema.registry.url}</param></schemaRegistryUrls>
    <subjects>
      <banking.account.debited-value>src/main/avro/.../MoneyDebited.avsc</banking.account.debited-value>
      <banking.account.credited-value>src/main/avro/.../MoneyCredited.avsc</banking.account.credited-value>
    </subjects>
  </configuration>
</plugin>
```

- Property `${schema.registry.url}` để ở parent pom, mặc định `http://localhost:8061` (**[LOCAL]** — prod override `-Dschema.registry.url=...` trong pipeline).
- **KHÔNG bind vào build thường** (tránh mỗi lần build lại push schema). Gọi goal thủ công/trong pipeline:
  - CI (chặn breaking change): `mvn io.confluent:kafka-schema-registry-maven-plugin:test-compatibility`
  - Deploy (đẩy schema): `mvn io.confluent:kafka-schema-registry-maven-plugin:register`

→ Đây là cách biến "lá chắn `409` của Registry" thành **bước bắt buộc trong CI** ("shift left", Bài 3.2 governance).
