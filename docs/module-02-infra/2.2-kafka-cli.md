# Bài 2.2 — Kafka CLI: tạo topic, produce/consume, quan sát lag

> Ghi lại **nguyên văn** nội dung buổi học: lý thuyết + thực hành + kết quả thật.
> Script: `source/infra/create-topics.sh`.

## 1. Bốn lệnh CLI cốt lõi

| Lệnh | Dùng để | Liên hệ lý thuyết |
|---|---|---|
| `kafka-topics` | Tạo/xóa/mô tả topic, partition, RF | 1.1, 1.2 |
| `kafka-console-producer` | Gõ tay message để gửi (test) | 1.3 |
| `kafka-console-consumer` | Đọc message từ topic (test) | 1.4 |
| `kafka-consumer-groups` | Xem **offset + lag** của group | 1.4 |

> 📌 Các lệnh nằm *bên trong* container Kafka (image Confluent), không cài trên Windows → chạy qua `docker exec kafka1 <lệnh>`. Bên trong container dùng listener INTERNAL `localhost:9092`.

## 2. ⭐ Tạo topic đúng thiết kế — "Topics as Code"

Prod-grade: **tắt auto-create**, tạo topic **tường minh** với đúng số partition/RF (Bài 1.1 Q&A). Trong prod, định nghĩa topic được quản như **code** (versioned, review được). Script `create-topics.sh` tạo 5 topic:

| Topic | Partitions | RF | cleanup.policy | retention |
|---|---|---|---|---|
| `banking.transfer.requested` | 6 | 3 | delete | 7 ngày |
| `banking.account.debited` | 6 | 3 | delete | 7 ngày |
| `banking.account.credited` | 6 | 3 | delete | 7 ngày |
| `banking.fraud.checked` | 3 | 3 | delete | 7 ngày |
| `banking.transfer.completed` | 3 | 3 | delete | 7 ngày |

Mỗi topic kèm `min.insync.replicas=2` (khẩu quyết 1.2). Topic audit/ledger thật sẽ để retention dài hơn nhiều.

Lệnh tạo (rút gọn):
```bash
docker exec kafka1 kafka-topics --bootstrap-server localhost:9092 \
  --create --if-not-exists --topic banking.account.debited \
  --partitions 6 --replication-factor 3 \
  --config min.insync.replicas=2 --config cleanup.policy=delete --config retention.ms=604800000
```

### ⚠️ Bài học từ WARNING khi tạo topic
> *"topics with a period ('.') or underscore ('_') could collide"* — khi xuất **metric**, Kafka thay cả `.` và `_` thành cùng ký tự → tên topic *vừa* có `.` *vừa* có `_` có thể làm metric đè nhau. Tên của ta **chỉ dùng `.`** → an toàn. Bài học: **đặt tên topic nhất quán một kiểu phân tách.**

## 3. ⭐ `--describe`: thấy tận mắt partition / leader / ISR

```
docker exec kafka1 kafka-topics --bootstrap-server localhost:9092 --describe --topic banking.account.debited
```
Kết quả thật:
```
PartitionCount: 6  ReplicationFactor: 3  Configs: min.insync.replicas=2,cleanup.policy=delete,retention.ms=604800000
  Partition 0   Leader: 3   Replicas: 3,1,2   Isr: 3,1,2
  Partition 1   Leader: 1   Replicas: 1,2,3   Isr: 1,2,3
  Partition 2   Leader: 2   Replicas: 2,3,1   Isr: 2,3,1
  Partition 3   Leader: 2   Replicas: 2,3,1   Isr: 2,3,1
  Partition 4   Leader: 3   Replicas: 3,1,2   Isr: 3,1,2
  Partition 5   Leader: 1   Replicas: 1,2,3   Isr: 1,2,3
```

| Quan sát | Khớp lý thuyết |
|---|---|
| 6 partition, mỗi cái **3 Replicas** trên 3 broker khác nhau | RF=3 (1.2) |
| **Leader rải đều**: broker 1/2/3 mỗi cái leader 2 partition | leader spread, cân tải (1.1) |
| **Isr = Replicas** (đủ 3) | cả 3 bản in-sync, cluster khỏe (1.2) |
| `Elr`/`LastKnownElr` rỗng | *Eligible Leader Replicas* (KRaft mới) — bình thường khi khỏe |

## 4. ⭐ Demo Key → Partition (lý thuyết 1.1 hiện ra)

Gửi 4 message với key, rồi đọc kèm partition:
```bash
printf 'acc-A:{...}\nacc-B:{...}\nacc-A:{...}\nacc-C:{...}\n' | \
  docker exec -i kafka1 kafka-console-producer --bootstrap-server localhost:9092 \
  --topic banking.account.debited --property parse.key=true --property key.separator=:

docker exec kafka1 kafka-console-consumer --bootstrap-server localhost:9092 \
  --topic banking.account.debited --from-beginning --max-messages 4 \
  --property print.key=true --property print.partition=true
```
Kết quả:
```
Partition:4  acc-C  {...}
Partition:0  acc-B  {...}
Partition:2  acc-A  {"amt":100}
Partition:2  acc-A  {"amt":30}
```
→ **CÙNG key `acc-A` → CÙNG partition 2 → giữ đúng thứ tự** (100 rồi 30). Key khác → rải partition khác → song song. Đúng nguyên tắc key = `accountId`. (Console consumer đọc nhiều partition nên thứ tự *toàn cục* không đảm bảo, nhưng *trong* acc-A thì đúng.)

## 5. ⭐⭐ Demo Consumer Lag (lý thuyết 1.4 hiện ra)

Quy trình: (B1) group `demo-ledger` đọc hết message hiện có → lag 0; (B2) bơm thêm 10 message key `acc-A` mà **không đọc**; (B3) xem lag:
```bash
docker exec kafka1 kafka-consumer-groups --bootstrap-server localhost:9092 --group demo-ledger --describe
```
Kết quả:
```
GROUP        TOPIC                    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG  CONSUMER-ID
demo-ledger  banking.account.debited  2          2               12              10   -
demo-ledger  banking.account.debited  0,1,3,4,5  ...             ...             0    -
```

**Ba điều quan trọng:**
1. **`LAG = LOG-END-OFFSET − CURRENT-OFFSET`** = 12 − 2 = 10 — đúng công thức 1.4.
2. ⭐ **Cả 10 message dồn vào Partition 2** vì cùng key `acc-A` → **hot partition** (cảnh báo 1.1 hiện ra thật: key lệch → một partition gánh hết).
3. **`CONSUMER-ID = "-"`** — group tồn tại nhưng **không có consumer đang chạy**; group vẫn giữ offset đã commit để lần sau đọc tiếp.

## 6. Nhìn trực quan trên Kafka UI (http://localhost:8080)
- **Topics** → `banking.account.debited`: 6 partition, Messages count, Replicas/ISR.
- **Consumers** → `demo-ledger`: cột **Lag** hiển thị trực quan, tập trung partition 2.
- Click partition → xem từng message (key, value, offset, timestamp).
