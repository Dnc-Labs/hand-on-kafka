## 2. ⭐ KRaft cần gì để chạy 3 broker (nhớ 1.1)

- **`KAFKA_NODE_ID`**: số định danh duy nhất (1, 2, 3).
- **`KAFKA_PROCESS_ROLES=broker,controller`**: mỗi node vừa là broker vừa tham gia bầu controller (KRaft, không ZooKeeper).
- **`KAFKA_CONTROLLER_QUORUM_VOTERS`**: danh sách **tất cả** controller để bầu quorum: `1@kafka1:9093,2@kafka2:9093,3@kafka3:9093`.
- **`CLUSTER_ID`**: một id chung cho cả cluster (cả 3 broker phải giống nhau).

## 3. ⭐⭐ Listeners — phần khó nhất và hay sai nhất

Một broker cần **nhiều "cửa" (listener)** vì có nhiều loại client kết nối tới nó theo các đường khác nhau:

| Listener | Ai dùng | Địa chỉ |
|---|---|---|
| **CONTROLLER** | Giao tiếp bầu controller giữa các node (KRaft) | `kafkaN:9093` |
| **INTERNAL** | Giao tiếp **giữa các container** (broker↔broker, Schema Registry, Kafka UI) | `kafkaN:9092` |
| **EXTERNAL** | Client **từ host** (Spring app của bạn) | `localhost:1909N` |

**Vì sao cần tách INTERNAL và EXTERNAL?**

> Khi một client kết nối, broker trả về **advertised listener** — địa chỉ để client *kết nối lại lần sau*. Một container khác trong network phải dùng tên `kafka1`; nhưng app chạy trên **host** không phân giải được `kafka1`, nó cần `localhost`. Cùng một broker phải "tự giới thiệu" bằng **hai địa chỉ khác nhau tùy người hỏi** → nên cần hai listener.

```
Schema Registry (trong network)  ──hỏi──▶ broker trả "kafka1:9092"    ✅ phân giải được
Spring app (trên host)           ──hỏi──▶ broker trả "localhost:19092" ✅ phân giải được
            (nếu chỉ có 1 listener trả "kafka1:9092" → app trên host KẾT NỐI HỎNG)
```

Đây là nguyên nhân #1 của lỗi *"connection refused / broker not available"* khi mới dựng Kafka bằng Docker.

**Mẹo kỹ thuật:** `KAFKA_LISTENERS` bind `0.0.0.0` (nghe mọi interface trong container), còn `KAFKA_ADVERTISED_LISTENERS` mới là địa chỉ "tự giới thiệu" tùy listener (INTERNAL→`kafka1`, EXTERNAL→`localhost`).

## 4. ⭐ Prod-grade vs Local (tách bạch rõ ràng)

**✅ Prod-grade (giữ nguyên khi lên thật):**
- `KAFKA_DEFAULT_REPLICATION_FACTOR=3`, `KAFKA_MIN_INSYNC_REPLICAS=2` (khẩu quyết 1.2).
- Replication topic nội bộ (`offsets`, `transaction-state`) = 3, min ISR = 2.
- `KAFKA_AUTO_CREATE_TOPICS_ENABLE=false` — tạo topic tường minh, không tự sinh.
- Schema Registry lưu schema với RF = 3.

**⚠️ Tiện ích local (KHÁC trên prod):**
- **`PLAINTEXT`**: prod bắt buộc **TLS + SASL/ACL** (Module 8.3). Rút gọn lớn nhất.
- **3 broker trên một máy**: không cách ly lỗi thật. Prod: 3 node/máy khác nhau, khác AZ.
- **`GROUP_INITIAL_REBALANCE_DELAY_MS=0`**: khởi động nhanh khi học. Prod để mặc định (3s).
- **Docker named volumes** trên máy bạn: prod dùng storage bền/định cỡ riêng.
- **Bản thân Docker Compose**: công cụ dev/staging. Prod: **Kubernetes + Strimzi** hoặc **managed** (Confluent Cloud, MSK).

> 📌 **Tư duy cấu hình (RF/ISR/no-auto-create) là thật, mang lên prod được; cách *đóng gói* (compose, plaintext, 1 máy) là local.**

## 5. Cổng truy cập từ host

| Dịch vụ | Địa chỉ | Dùng để |
|---|---|---|
| Kafka broker 1/2/3 | `localhost:19092` / `19093` / `19094` | bootstrap server cho app |
| Schema Registry | http://localhost:8081 | quản Avro schema |
| Kafka UI | http://localhost:8080 | nhìn trực quan |

App trên host dùng: `bootstrap.servers = localhost:19092,localhost:19093,localhost:19094`

## 6. Lệnh vận hành

```bash
docker compose up -d            # khởi động (nền)
docker compose ps               # trạng thái + healthcheck
docker compose logs -f kafka1   # log một broker
docker compose down             # dừng (giữ dữ liệu)
docker compose down -v          # dừng + XÓA SẠCH dữ liệu (reset)
```
