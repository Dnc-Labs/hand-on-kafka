# Bài 1.5 — Delivery Semantics & Transactions

> Ghi lại **nguyên văn** nội dung buổi học: lý thuyết + câu hỏi của học viên + câu trả lời.

Đây là **đỉnh lý thuyết của Module 1**. Mọi thứ ta rải suốt 1.2–1.4 (acks, idempotent producer, offset commit) hội tụ ở đây để trả lời: *"Làm sao đảm bảo mỗi message được xử lý **đúng một lần**?"* Và quan trọng hơn: **exactly-once thật sự nghĩa là gì, và nó KHÔNG có nghĩa là gì.**

## 1. Ba mức Delivery Semantics — định nghĩa chuẩn

| Semantics | Nghĩa | Hệ quả |
|---|---|---|
| **At-most-once** | Mỗi message xử lý **0 hoặc 1 lần** | Có thể **mất**, không bao giờ trùng |
| **At-least-once** | Mỗi message xử lý **1 hoặc nhiều lần** | Không bao giờ mất, có thể **trùng** |
| **Exactly-once** | Mỗi message xử lý **đúng 1 lần** | Không mất, không trùng — lý tưởng, khó nhất |

Ta đã gặp at-most/at-least ở 1.4 (qua thứ tự commit-vs-process). Bài này tập trung vào **exactly-once**.

## 2. Vì sao Exactly-Once khó? — Bài toán có HAI phía

Trong luồng **consume → process → produce**, phải đảm bảo **đồng thời hai thứ**, nguyên tử cùng nhau:

```
        đọc event A          ghi event B (kết quả)
Consumer ─────────▶ process ─────────▶ Producer ─▶ Kafka
   │                                              │
   └─ commit offset của A ◀──── phải cùng "thành công hoặc thất bại" ────┘
      (để không đọc lại A)         với việc ghi B
```

Hai mảnh dễ vỡ:
1. **Phía producer**: ghi B có thể bị **trùng** do retry (1.3) → idempotent producer giải quyết.
2. **Phía consumer**: ghi B thành công nhưng **crash trước khi commit offset A** → restart đọc lại A → **ghi B lần nữa**. Ngược lại, commit offset A trước nhưng crash trước khi ghi B → **mất B**.

→ Cốt lõi: **"ghi B" và "commit offset A" là hai thao tác riêng**, ở hai nơi (topic kết quả vs `__consumer_offsets`). Không nguyên tử → luôn có khe hở crash. Idempotent producer **một mình không đủ** vì không bao luôn offset commit.

## 3. ⭐ Kafka Transactions — biến nhiều thao tác thành MỘT đơn vị nguyên tử

Cho phép **ghi nhiều message (nhiều partition/topic) + commit offset** trong **một transaction** — hoặc tất cả commit, hoặc tất cả abort. Công cụ là **transactional producer** (`transactional.id` — Bài 1.3):

```java
producer.initTransactions();                      // 1 lần lúc khởi tạo
while (true) {
    records = consumer.poll(...);
    producer.beginTransaction();                  // ── mở transaction
    try {
        for (record : records) {
            ResultB b = process(record);
            producer.send(topicB, b);             // ghi event kết quả
        }
        // ⭐ commit offset CỦA CONSUMER nằm LUÔN trong transaction của producer
        producer.sendOffsetsToTransaction(offsetsOf(records), consumer.groupMetadata());
        producer.commitTransaction();             // ── tất cả cùng commit nguyên tử
    } catch (Exception e) {
        producer.abortTransaction();              // ── tất cả cùng hủy
    }
}
```

Điểm thiên tài: **`sendOffsetsToTransaction`** đưa việc *commit offset của consumer* vào *transaction của producer* → "ghi B" và "đánh dấu đã-đọc-A" thành **một** → không còn khe hở crash. Đây là **consume-transform-produce** với EOS.

Cơ chế nền (1.3): `transactional.id` → danh tính ổn định qua restart + **zombie fencing** (epoch). Broker dùng **Transaction Coordinator** + **transaction log** theo dõi trạng thái.

## 4. ⭐ `isolation.level` — phía consumer phải đọc đúng cách

| `isolation.level` | Consumer thấy gì |
|---|---|
| **`read_uncommitted`** (mặc định) | Thấy **mọi** message, kể cả transaction **chưa commit / sẽ bị abort** → đọc nhầm dữ liệu "ma" |
| **`read_committed`** ⭐ | **Chỉ thấy** message của transaction **đã commit**; message bị abort bị **ẩn đi** |

```
Topic B:  [committed][committed][ABORTED][in-progress][committed]
read_committed consumer chỉ đọc:  ▲         ▲                      ▲
                                  (bỏ qua ABORTED và in-progress)
```

> 💡 Với banking dùng transactions thì consumer phía sau **bắt buộc** `read_committed`. (Chi tiết: `read_committed` chỉ đọc tới **LSO — Last Stable Offset**, có thể tăng nhẹ latency vì chờ transaction đóng. Đánh đổi đáng giá.)

## 5. ⭐⭐ SỰ THẬT QUAN TRỌNG NHẤT: EOS của Kafka chỉ trong biên giới Kafka

> **Exactly-once semantics (EOS) của Kafka chỉ đảm bảo cho luồng Kafka → Kafka** (đọc topic, xử lý, ghi topic + commit offset). Nó **KHÔNG** tự mở rộng ra **hệ thống bên ngoài** như database, REST call, gửi email.

Vì transaction của Kafka chỉ bao những thứ *trong Kafka* (topic + `__consumer_offsets`). Ghi vào **DB ngoài** hay gọi **API ngoài** → không nằm trong transaction Kafka → không rollback được.

```
✅ Kafka EOS bao được:   đọc topic A  →  ghi topic B  +  commit offset      (tất cả trong Kafka)
❌ Kafka EOS KHÔNG bao:  đọc topic A  →  UPDATE balance trong Postgres       (DB nằm ngoài)
                                      →  gọi cổng thanh toán (API ngoài)
```

→ Trong banking, hành động chính là **"trừ/cộng tiền trong DB"** — **nằm ngoài Kafka**. Nên **không thể chỉ bật `transactional.id` rồi tuyên bố "exactly-once"**. Vì vậy cần:

| Bài toán | Giải pháp | Module |
|---|---|---|
| Ghi DB + phát event nguyên tử | **Transactional Outbox** | 6.1 |
| Consumer xử lý lại không gây tác dụng kép lên DB | **Idempotent consumer** (idempotency key) | 5.4 |
| EOS thuần Kafka→Kafka | **Kafka transactions / Kafka Streams EOS** | 6.5 / 7 |

> 💡 *"Exactly-once của Kafka là **end-to-end trong Kafka**, không phải **end-to-end trong hệ thống của bạn**. Khi có database, **exactly-once thực dụng = at-least-once + idempotency**."*

## 6. Bức tranh tổng hợp — ráp mọi mảnh của Module 1

```
                 ĐỘ BỀN              KHÔNG TRÙNG/ĐẢO          NGUYÊN TỬ            ĐỌC ĐÚNG
                (1.2)               (1.3)                   (1.5)               (1.5)
Producer:    acks=all          +  enable.idempotence  +  transactional.id          —
             min.insync=2          (PID+sequence)          (+ sendOffsets)
                                                              │
Consumer:    manual commit                                   │            +  isolation.level=
             sau xử lý (1.4)   ──────────────────────────────┘               read_committed
             + idempotency nghiệp vụ (5.4) cho phần chạm DB ngoài
```

- **at-least-once** (mục tiêu mặc định của banking) = `acks=all` + idempotent producer + manual commit sau xử lý. **Đủ dùng** cho hầu hết luồng, *với điều kiện* consumer idempotent.
- **exactly-once (Kafka→Kafka)** = thêm transactions + `read_committed`. Dùng cho stream nội bộ.
- **exactly-once thực tế khi có DB** = at-least-once + **Outbox** + **idempotent consumer**. Con đường chính của dự án.

## 7. Dự án Banking chọn gì? (định hướng)
- **Mặc định toàn hệ**: **at-least-once + idempotency** — đơn giản, bền, đủ đúng. Lựa chọn của đa số hệ tài chính thật.
- **Transactions/EOS**: dùng chọn lọc ở **Outbox relay** và **Kafka Streams** (tổng hợp ledger/fraud), nơi luồng thực sự Kafka→Kafka.
- **Không** ép exactly-once toàn cục bằng transactions cho mọi thứ — vừa chậm, vừa không bao DB ngoài, vừa phức tạp.

---

📌 **Tóm tắt Bài 1.5:**
- Ba semantics: **at-most-once** (mất), **at-least-once** (trùng), **exactly-once** (đúng một lần).
- Exactly-once khó vì consume→process→produce có **hai phía** phải nguyên tử cùng nhau: ghi kết quả + commit offset.
- **Kafka transactions** (`transactional.id`, `beginTransaction`/`commitTransaction`, **`sendOffsetsToTransaction`**) gộp ghi-nhiều-message + commit-offset thành **một đơn vị nguyên tử**; consumer đọc bằng **`read_committed`**.
- **SỰ THẬT cốt tử: EOS của Kafka chỉ trong biên giới Kafka.** Không tự bao DB/API ngoài.
- Khi có DB (case banking): **exactly-once thực dụng = at-least-once + Outbox + idempotent consumer**.
- Dự án ta: mặc định **at-least-once + idempotency**; transactions/EOS dùng chọn lọc cho Outbox relay & Kafka Streams.

# Hỏi đáp Bài 1.5

## Câu hỏi: Outbox với transactions khác nhau thế nào, sao không dùng transactions luôn cho DB?

Câu hỏi này chạm thẳng vào **dual-write problem**, bài toán trung tâm của event-driven microservices.

### 1. Outbox và Kafka transactions giải HAI bài toán KHÁC NHAU

| | **Kafka transactions** | **Outbox pattern** |
|---|---|---|
| Nguyên tử **giữa cái gì** | Nhiều topic Kafka + offset — **toàn bộ trong Kafka** | **DB (state) ↔ phát event ra Kafka** — bắc cầu 2 hệ |
| Bài toán giải | consume → transform → produce **trong Kafka** | **dual-write**: vừa đổi DB vừa phát event |
| Phạm vi | Kafka ↔ Kafka | DB ↔ Kafka |

→ Không phải hai cách làm cùng một việc. Outbox giải đúng cái Kafka transactions **không thể** chạm: **database nằm ngoài Kafka**.

### 2. Bài toán dual-write — vì sao khó
Account Service "trừ tiền" cần làm **hai việc ở hai hệ**:
```
1. UPDATE balance SET amount = amount - 1tr   (Postgres)
2. publish event "MoneyDebited"               (Kafka)
```
Hai hệ **không chia sẻ transaction**. Mọi cách ghép ngây thơ đều có khe hở crash:
```
A: ghi DB xong rồi publish:  UPDATE ✅ ──CRASH─✖ publish chưa kịp → tiền trừ nhưng KHÔNG có event ❌
B: publish xong rồi ghi DB:  publish ✅ ──CRASH─✖ UPDATE chưa kịp → event ma, DB chưa trừ ❌
```
**`@Transactional` KHÔNG cứu được** — nó chỉ bao transaction *database*; `kafkaTemplate.send()` **không nằm trong** transaction đó.

### 3. Vì sao KHÔNG dùng Kafka transactions cho DB được (trả lời thẳng)
> **Kafka transaction về kỹ thuật KHÔNG THỂ bao gồm thao tác database.** Không có API nào ghi danh một `UPDATE` Postgres vào transaction Kafka. Transaction Kafka chỉ biết resource *trong Kafka* (topic + `__consumer_offsets`).

Thứ duy nhất "một transaction ôm cả Postgres lẫn Kafka" là **2PC/XA** — đã bác bỏ ở **Bài 0.4**: Kafka không hỗ trợ đáng tin; 2PC có blocking lock, coordinator SPOF, coupling chặt, throughput thấp. → Giới công nghiệp né 2PC, dùng **Outbox**.

### 4. ⭐ Ý tưởng cốt lõi của Outbox: biến "2 hệ" thành "1 hệ"
> Thay vì ghi vào **hai** hệ (không nguyên tử được), chỉ ghi vào **một** hệ (DB) nguyên tử, rồi *sau đó* chuyển ra Kafka.

Trong **cùng một local transaction DB** (ACID thật), ghi cả hai:
```
┌──────────── MỘT local DB transaction (ACID) ────────────┐
│  UPDATE balance SET amount = amount - 1tr                │
│  INSERT INTO outbox (event_type, payload, aggregate_id)  │   ← cùng commit hoặc cùng rollback
│         VALUES ('MoneyDebited', {...}, 'accountA')        │
└──────────────────────────────────────────────────────────┘
                         │ (đã commit an toàn — không thể mất 1 nửa)
                         ▼
              ┌─────────────────────┐
              │  Message Relay      │   tiến trình riêng: đọc bảng outbox,
              │  (Outbox publisher) │   publish lên Kafka, đánh dấu đã gửi
              └──────────┬──────────┘
                         ▼
                      [ KAFKA ]  ──▶ Ledger / Notification / ...
```
Bước 1+2 trong **một** transaction của **một** DB → **nguyên tử thật** (DB lo, không cần 2PC). Khe hở crash biến mất.

### 5. Bước relay ra Kafka — không lại dual-write à?
Relay là **at-least-once**, và idempotency vào cuộc:
- Relay đọc outbox row **chưa gửi**, publish, đánh dấu `sent=true`.
- Crash *sau publish, trước đánh dấu* → publish **lại** → **event trùng**. **Chấp nhận được** vì consumer phía sau **idempotent** (5.4) và/hoặc relay dùng **idempotent producer** (1.3).

→ Outbox **không xóa** trùng lặp; nó **dời** vấn đề từ "mất/sai dữ liệu" (không chữa được) sang "trùng lặp" (chữa được). Đúng tinh thần 1.5: *exactly-once thực dụng = at-least-once + idempotency*.

### 6. Hai kiểu relay
| Kiểu | Cách làm | Ưu / Nhược |
|---|---|---|
| **Polling Publisher** | Job định kỳ `SELECT * FROM outbox WHERE sent=false` → publish → update | Đơn giản, tự code; nhược: độ trễ poll, tải DB |
| **CDC / Log Tailing** (**Debezium**) | Đọc thẳng **transaction log DB** (WAL/binlog), bắt INSERT vào outbox, đẩy lên Kafka | Thanh lịch, low-latency, không cần app poll; nhược: thêm hạ tầng |

> 💡 CDC (Debezium) là cách "đỉnh" prod (không động app, đọc commit log DB). Ta làm **Polling Publisher trước** (hiểu bản chất), rồi *có thể* giới thiệu Debezium ở Module 6.

### 7. Kafka transactions còn dùng ở đâu? — BỔ TRỢ nhau, không loại trừ
- **Outbox** giải **DB → Kafka** (phát event từ service có database).
- **Kafka transactions** giải **Kafka → Kafka** (stream đọc-xử lý-ghi + commit offset nguyên tử — Kafka Streams, Module 7).
- Bản thân relay có thể dùng idempotent/transactional producer. Chúng **xếp tầng**, mỗi cái lo một đoạn:
```
[Account DB] ──Outbox──▶ [Kafka] ──Kafka tx/Streams EOS──▶ [Kafka] ──▶ [Ledger DB] (idempotent consumer)
   nguyên tử DB           ↑ at-least-once       ↑ EOS Kafka→Kafka     ↑ idempotency cho DB ngoài
```

### 8. Bảng chốt
| Câu hỏi | Outbox | Kafka transactions |
|---|---|---|
| Bao được DB không? | **Có** (qua local DB tx) | **Không** |
| Bao nhiều topic + offset nguyên tử? | Không | **Có** |
| Dùng 2PC/XA? | **Không** (chỉ local tx) | Không (cơ chế nội bộ Kafka) |
| Giải bài toán | **dual-write DB↔Kafka** | **consume-transform-produce trong Kafka** |
| Vị trí trong dự án | Account/Transaction phát event (M6.1) | Kafka Streams tổng hợp (M6.5, M7) |

> 💡 *Outbox để **phát event từ một service có database** mà không mất/sai. Kafka transactions để **nối các bước trong Kafka**. Cái cần khi có DB là **Outbox**, không phải Kafka transactions — vì Kafka transactions không với tới DB.*
